3. NEW BUGS FOUND

BUG-NEW-1 [CRITICAL]: Adaptive Slab Sizing Too Aggressive for 1024B+ Blocks
----------------------------------------------------------------------------
File: global_heap.cpp, lines 31-33

  std::size_t target_blocks = 256;
  if (block_size >= 1024) target_blocks = 64;
  if (block_size >= 4096) target_blocks = 16;

With only 16 blocks per slab for 4096B and 64 for 1024B, the FreeOnly and
RandomSize benchmarks create far too many slabs. Here's the math:

For FreeOnly 4096B (batch=500 blocks):
  - TLS cache max = 64 blocks
  - Remaining 436 blocks need 436/16 = ~28 slabs
  - Each iteration creates 28 slabs via VirtualAlloc
  - Each free iteration processes returns across 28 slabs
  - Per-slab overhead: lock acquisition, DrainPendingReturns, RemoveSlab, etc.

For RandomSize 8T (batch=500 random blocks 1-4096B):
  - Hits ~256 different size classes
  - Many size classes in 1024-4096B range create new small slabs
  - 8 threads on 4 cores = massive contention on the 16 striped locks
  - The deferred return queues overflow with returns to many different slabs

EVIDENCE:
  FreeOnly 4096B/1T: 209263 ns (17x slower than std)
  FreeOnly 1024B/4T: 273438 ns (16x slower than std)
  RandomSize 8T: 5208333 ns (60x slower than std)

FIX: Increase target_blocks for the 1024B+ tiers:

  std::size_t target_blocks = 256;
  if (block_size >= 1024) target_blocks = 128;   // was 64
  if (block_size >= 4096) target_blocks = 32;    // was 16

This gives:
  1024B: 128 blocks * 1024B = 128KB slabs (was 64KB)
  4096B: 32 blocks * 4096B = 128KB slabs (was 64KB)

Still a 75% memory savings vs the original 256-block slabs (1MB for 4096B),
but provides enough blocks per slab to avoid thrashing.

IMPACT: Eliminates all 5 FreeOnly large-size losses and the RandomSize 8T
catastrophe. Estimated to restore FreeOnly 4096B to ~2-5x faster than std
(based on how 256B performs with similar slab density).

EFFORT: 2 lines changed. RISK: None (only affects slab size, not correctness).


BUG-NEW-2 [MEDIUM]: SlabToFree Stack Buffer Is Now 2KB on Stack
-----------------------------------------------------------------
File: global_heap.h, line 86; global_heap.cpp, lines 91, 145, 157, 193, 205

  static constexpr std::size_t MAX_DEFERRED_FREE = 128;
  struct SlabToFree {
      void* ptr;        // 8 bytes
      std::size_t size; // 8 bytes
  };
  // 16 bytes per entry * 128 = 2048 bytes

  SlabToFree stack_buf[MAX_DEFERRED_FREE];  // 2KB on stack

This appears in 4 functions: AllocateBlock, AllocateBatch, DeallocateBlock
(not directly but DeallocateBatch uses its own stack_buf), and DeallocateBatch.

On Windows, the default stack size is 1MB per thread, so 2KB is fine.
However, AllocateBlock and AllocateBatch each allocate 2KB on stack inside
the spinlock scope. Under deep call stacks, this could contribute to stack
pressure.

FIX: This is acceptable for now. If stack usage becomes a concern, consider
allocating from a thread-local buffer instead:

  thread_local SlabToFree t deferred_buf[MAX_DEFERRED_FREE];

EFFORT: Low. RISK: None.


BUG-NEW-3 [LOW]: DeallocateBatch Does Unconditional exchange After process_list
--------------------------------------------------------------------------------
File: global_heap.cpp, line 232

  FreeBlock* pending = pending_returns_[stripe].head.exchange(nullptr, ...);
  process_list(pending);

Even though DrainPendingReturns (line 63-64) now has a relaxed-load fast path,
the DeallocateBatch function still does an unconditional exchange on the
pending list after processing the direct head list. This exchange always
happens, even if the pending list was already drained by DrainPendingReturns
earlier in the same call.

However, this is harder to fix because another thread might have pushed to
the pending list between the DrainPendingReturns call and this exchange.
The exchange here is intentional — it ensures we process any returns that
arrived between our initial drain and now. The overhead is one atomic exchange
per DeallocateBatch call, which is acceptable.

FIX: None needed. The behavior is correct. The exchange is necessary for
completeness — it catches returns that were deferred by other threads while
we were processing the direct list.

OPT-NEW-2 [MEDIUM]: Reduce SpinLock Yield Threshold from 32 to 64
--------------------------------------------------------------------
File: global_heap.h, line 20

  if (++spins > 32) {

On x86, a spin-wait loop with _mm_pause() is extremely efficient (avoids
pipeline stalls and has near-zero overhead). The current threshold of 32
means we yield to the OS scheduler after only 32 spins (~100ns at 2.4GHz).

On your 4-core machine, HeavyContention is already winning 3-6x, so the
spinlock is not a bottleneck. But yielding too early adds unnecessary
scheduler overhead and can cause context switches that slow down the critical
path.

FIX: Increase to 64 or 128:
  if (++spins > 64) {

This is a minor tuning knob. The current value works well for your hardware
(HeavyContention is dominant), so this is low priority.


OPT-NEW-3 [LOW]: Prefetch First Batch Element Too
-------------------------------------------------
File: tls_cache.cpp, lines 112-117

  FreeBlock* curr = batch_head->next;
  for (int i = 0; i < 4 && curr; ++i) {

The current prefetch loop starts from batch_head->next, skipping batch_head
itself. Since batch_head is about to be returned to the caller, the caller
will immediately access it. Prefetching batch_head would reduce the latency
of the first access.

FIX:
  FreeBlock* curr = batch_head;
  for (int i = 0; i < 4 && curr; ++i) {
      __builtin_prefetch(curr, 0, 1);
      curr = curr->next;
  }

Expected improvement: Marginal (~1-2ns per cache miss avoided).

MEM-NEW-2 [LOW]: FreeBlock Padding Could Be Reduced for Large Blocks
----------------------------------------------------------------------
File: slab.h, lines 11-15

  struct FreeBlock {
      Slab* slab;                        // 8 bytes
      char _padding[16 - sizeof(Slab*)]; // 8 bytes wasted
      FreeBlock* next;                   // 8 bytes (overwritten by user)
  };

Total FreeBlock size: 24 bytes with 16-byte alignment padding.

The _padding exists to ensure the user data starts at offset 16 from the
block base, preserving 16-byte SIMD alignment. However, for block sizes
that are already multiples of 16 (which ALL of them are, since ALIGNMENT=16),
the slab pointer at offset 0 is the only metadata that needs protection.

This is the same as the OPT-11 discussion we agreed to skip. The 16-byte
offset is the safe choice for SIMD alignment. No change recommended.

CORR-1 [LOW]: Add Debug Assertions for Slab Invariants
--------------------------------------------------------
Currently, the Slab struct has no runtime validation. Adding debug assertions
would catch use-after-free, double-free, and slab corruption earlier:

  void Slab::Deallocate(void* ptr) {
      assert(ptr != nullptr);
      FreeBlock* block = static_cast<FreeBlock*>(ptr);
      assert(block->slab == this && "Block returned to wrong slab!");
      assert(free_blocks < total_blocks && "Double free detected!");
      block->next = free_list;
      free_list = block;
      free_blocks++;
  }

In slab.cpp, add to Allocate():
  void* Slab::Allocate() {
      assert(free_blocks > 0 && free_list != nullptr);
      ...
  }

These should be guarded by #ifndef NDEBUG to have zero overhead in Release.

CORR-2 [LOW]: fast_realloc(nullptr, 0) is Implementation-Defined
-------------------------------------------------------------------
File: fast_alloc.cpp, lines 77-84

  if (new_size == 0) {
      fast_free(ptr);
      return nullptr;
  }
  if (!ptr) {
      return fast_malloc(new_size);
  }

If ptr is null AND new_size is 0: the first check skips (ptr is null,
fast_free(nullptr) is a no-op), returns nullptr. This is correct.
But if new_size is 0 with a valid ptr, we free and return nullptr. Per the
C standard, this behavior is implementation-defined. MSVC's free(nullptr)
is a no-op, so this is safe. Just noting for documentation purposes.

7. ARCHITECTURE OBSERVATIONS
================================================================================

OBS-1: The Global Heap Has No Slab Count Limits
------------------------------------------------
The partial_slabs_ and full_slabs_ arrays have no upper bound on how many
slabs each size class can accumulate. Under sustained mixed workloads
(where blocks are allocated and freed across many slabs), the number of
partial slabs can grow without bound.

Currently this is mitigated by:
  - MAX_DEFERRED_FREE = 128 (empty slabs are freed when limit reached)
  - Slabs become full and move to full_slabs_ (reducing partial list)
  - Full slabs move back to partial on free (which can then be freed if empty)

In practice, this self-regulates. But under pathological workloads (e.g.,
thread A allocates from slab 1, thread B from slab 2, both free to their
own slabs, creating many partially-used slabs), memory usage could grow.

Priority: LOW (not observed in any benchmarks).


OBS-2: The TLS Cache Has 512 Size Class Entries
-----------------------------------------------
The TLSCache has two arrays of 512 entries each:
  fast_bins_:  512 * 8 = 4096 bytes
  counts_:     512 * 8 = 4096 bytes
  Total: 8192 bytes per thread

On your Windows machine with 32KB L1 data cache, the TLS cache occupies
25% of L1. When a thread accesses fast_bins_[class_index], it loads the
pointer from the array (likely an L1 hit since the thread uses only a few
size classes repeatedly). This is acceptable but worth noting.

A potential optimization (FAR FUTURE): use a hash map for the upper size
classes (4096-8192) since they're rarely used. This would shrink the cache
from 8KB to ~5KB, improving L1 density for the hot small-size path.

Priority: VERY LOW (not a bottleneck in any benchmark).
